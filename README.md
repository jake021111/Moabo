# My_multi_algorithm
05.14
프로젝트 개요: 키워드를 통해 여러 미디어의 알고리즘을 모아 한번에 볼수 있게 추천해주는 서비스

개발동기: 바이브 코딩으로 소프트웨어를 만들라는 과제를 통해 내가 무엇을 필요로 할까 생각해 본 결과
평소에 여러 ott, 게임, 유튜브 등을 하거나 볼때마다 각각의 앱에 들어가서 찾기 내 취향에 맞는 여러 작품들을 찾기 귀찮았는데 
하나의 사이트에서 다 해결 할 수 있다면 내가 쓰기에도 편할 것 같아서 제작하자고 합니다.



05.19
기능적 요구사항
1. 회원 및 취향 관리
사용자 인증: 사용자는 간단한 계정 생성을 통해 로그인을 수행할 수 있어야 한다.
취향 키워드 등록/수정: 사용자는 자신이 선호하는 핵심 키워드(예: #디스토피아, #메카물, #오픈월드 등)를 자유롭게 등록하고 관리할 수 있다.
카테고리  설정: 사용자는 4가지 도메인(영화/드라마, 애니, 게임, 유튜브) 중 선호하는 카테고리만 선택해서 보여지게 할 수 있다

3. 멀티미디어 데이터 연동 및 정제
외부 API 데이터 수집: 시스템은 백엔드에서 다음 4가지 외부 오픈 API와 연동한다. (영화/드라마: TMDB API, 애니메이션: Jikan API, 게임: RAWG API, 영상: YouTube Data API v3)
데이터 규격 표준화: 각기 다른 형식의 외부 JSON 데이터를 백엔드에서 공통 DTO 구조(CommonContentDto)로 변환 및 정제한다.

4. 통합 알고리즘 큐레이션 및 검색
키워드 기반 통합 피드: 메인 화면에서 사용자가 등록한 취향 키워드를 종합하여 4개 도메인의 추천 콘텐츠가 믹스된 '통합 피드'를 제공한다.
실시간 통합 검색: 검색창에 특정 키워드 입력 시, 외부 API들을 동시에 호출하여 카테고리별 검색 결과를 한눈에 보여준다.
콘텐츠 상세 조회: 특정 콘텐츠를 선택하면 줄거리, 평점, 출시일, 관련 유튜브 예고편 링크 등 상세 정보를 제공한다.

5. 북마크 관리
Vibe 보관함 (찜): 마음에 드는 콘텐츠는 도메인 구분 없이 하나의 보관함에 저장할 수 있다.

비기능적 요구사항
1. 성능 및 안정성
비동기 멀티 스레딩 호출: 4개의 외부 API를 동기 방식으로 순차 호출할 경우 지연 시간이 너무 길어지므로, 백엔드(Spring Boot)에서 CompletableFuture 등을 활용해 비동기 병렬 호출을 수행하여 응답 속도를 2초 이내로 유지한다.
캐싱(Caching) 전략: 자주 검색되는 키워드나 변경 주기가 긴 데이터(예: 영화 정보)는 로컬 캐시를 적용하여 외부 API 호출 트래픽을 절감한다.

2. 확장성 및 유지보수성
인터페이스 추상화: 추후 새로운 콘텐츠 소스(예: 도서, 웹소설 API)가 추가되더라도 기존 서비스 로직을 건드리지 않고 확장할 수 있도록 인터페이스 중심 설계를 적용한다.

3. 보안
API Key 은닉: 외부 API 호출에 필요한 인증 Key 파일은 GitHub 소스코드에 직접 노출하지 않고, 시스템 환경변수(env) 또는 application.yml 외부 지정을 통해 관리한다.


05.20
유스케이스 및 작동 시나리오
1. 유스케이스 정의
유스케이스 모델은 사용자(이태균)가 시스템(My_multi_algorithm)과 상호작용하여 얻고자 하는 핵심 서비스 흐름을 정의합니다.

액터: 사용자 (개인 취향 멀티미디어를 소비하고자 하는 주체)
시스템: My_multi_algorithm 백엔드 및 프론트엔드 서비스

핵심 유스케이스 목록:
취향 키워드 설정: 사용자가 자신의 선호 태그를 등록 및 관리하는 기능
통합 콘텐츠 피드 조회: 등록된 키워드 기반의 믹스된 콘텐츠 리스트를 확인하는 기능
실시간 멀티 도메인 검색: 특정 키워드로 4개 API 데이터를 동시에 조회하는 기능
콘텐츠 보관: 도메인에 관계없이 원하는 작품을 저장하는 기능

2. 작동 시나리오 (Use Case Scenarios)
취향 키워드 설정 시나리오
기본 흐름:
사용자가 앱의 프로필 또는 설정 메뉴로 진입한다.
사용자가 자신의 취향 키워드(예: #디스토피아, #오픈월드)를 입력창에 추가한다.
사용자가 각 미디어 카테고리(영화, 애니, 게임, 유튜브)의 선호 가중치를 슬라이더로 조절한다.
사용자가 저장 버튼을 누르면 시스템은 해당 데이터를 사용자의 로컬 또는 DB 세션에 반영한다.

예외 흐름:
키워드가 공백이거나 특수문자만 포함된 경우 시스템은 경고 메시지를 출력하고 저장을 중단한다.

통합 콘텐츠 피드 조회 시나리오 (핵심 작동 로직)
기본 흐름:
사용자가 앱의 메인 화면에 진입한다.
시스템은 사용자가 설정한 취향 키워드 목록을 확인한다.
백엔드 서버는 설정된 키워드를 기반으로 4개의 외부 오픈 API(TMDB, Jikan, RAWG, YouTube)에 비동기적으로 동시에 데이터 요청을 보낸다.
각 API로부터 받아온 서로 다른 형태의 데이터를 공통 규격으로 정제 및 매핑한다.
사용자가 설정한 도메인별 가중치를 적용하여 데이터를 믹싱(Mixing)하고 정렬한다.
메인 피드 화면에 영화, 애니, 게임, 관련 유튜브 영상이 교차되어 배열된 통합 추천 리스트를 사용자에게 보여준다.

예외 흐름:
특정 API(예: 게임 API)의 응답이 지연되거나 에러가 발생한 경우, 시스템은 해당 카테고리를 제외하고 정상 응답이 온 나머지 3개 카테고리의 데이터만 정제하여 피드를 구성한다. 전체 시스템이 멈추지 않도록 방어적인 처리를 수행한다.

실시간 멀티 도메인 검색 시나리오
기본 흐름:
사용자가 메인 화면 상단의 통합 검색창에 검색어(예: 귀멸의 칼날)를 입력한다.
사용자가 검색 버튼을 누른다.
시스템은 입력된 검색어를 검색 쿼리로 삼아 4개 도메인의 오픈 API를 실시간 병렬 호출한다.
각 도메인별 검색 결과를 카테고리 탭(영화/애니/게임/유튜브) 또는 하나의 통합 섹션으로 분할하여 실시간으로 화면에 출력한다.
사용자가 특정 결과물을 클릭하면 해당 콘텐츠의 상세 정보 조회 유스케이스로 이어진다.

5.27
아키텍처 설계서
1. 시스템 구조 (System Architecture)
본 시스템은 외부 미디어 API의 데이터를 백엔드에서 취합 및 정제하여 프론트엔드로 전달하는 API 엔지니어링 중심의 구조를 가집니다.

Client Layer (Frontend)
사용자가 미디어 피드를 소비하고 검색하는 인터페이스
기술 스택: Flutter 또는 React (사용자 환경에 맞게 유연하게 구성)

Application Layer (Backend)
프론트엔드의 요청을 받아 외부 API를 제어하고 데이터를 가공하는 핵심 엔진
기술 스택: Java, Spring Boot, Spring WebFlux (또는 WebClient)

Data & External Layer
시스템 데이터 관리 및 외부 미디어 소스 공급처
내부 저장소: MySQL
외부 소스: TMDB API (영화/드라마), Jikan API (애니), RAWG API (게임), YouTube Data API (영상)

2. 데이터 흐름 (Data Flow)
사용자 취향 조회: 메인 진입 시 내부 MySQL에서 사용자의 관심 키워드 및 가중치 조회
멀티 도메인 요청: 수집된 키워드를 파라미터로 삼아 4개의 외부 오픈 API에 병렬로 비동기 요청 송신
데이터 정제 및 믹싱: 외부 API의 응답 데이터(JSON)를 공통 포맷으로 변환한 뒤, 가중치 알고리즘에 따라 데이터를 결합하고 정렬
클라이언트 반환: 정제된 단일 통합 리스트를 프론트엔드로 전송

ERD (Entity Relationship Diagram)
텍스트로 표현한 논리적 관계도입니다. 테이블 간의 연결 관계를 나타냅니다.

[User (사용자)]
1 : N 관계 -> [UserKeyword (사용자 취향 키워드)]
1 : N 관계 -> [Bookmark (Vibe 보관함)]

[UserKeyword]
각 사용자가 등록한 관심 태그 목록

[Bookmark]
사용자가 찜한 외부 콘텐츠의 고유 식별자 및 카테고리 정보 보관

DB 설계서
MySQL 기준 테이블 상세 명세입니다.

1. User 테이블 (사용자 정보)
Column: id / Type: BIGINT / Key: PK / Extra: Auto Increment / 설명: 사용자 고유 식별자
Column: username / Type: VARCHAR(50) / Key: Unique / 설명: 로그인 아이디
Column: password / Type: VARCHAR(255) / 설명: 암호화된 비밀번호
Column: weight_movie / Type: INT / 설명: 영화 노출 가중치 (0~100)
Column: weight_anime / Type: INT / 설명: 애니 노출 가중치 (0~100)
Column: weight_game / Type: INT / 설명: 게임 노출 가중치 (0~100)
Column: weight_youtube / Type: INT / 설명: 유튜브 노출 가중치 (0~100)

2. UserKeyword 테이블 (사용자 취향 키워드)
Column: id / Type: BIGINT / Key: PK / Extra: Auto Increment / 설명: 키워드 식별자
Column: user_id / Type: BIGINT / Key: FK (User.id) / 설명: 참조하는 사용자 식별자
Column: keyword_name / Type: VARCHAR(50) / 설명: 취향 키워드명 (예: 디스토피아, 루프물)

3. Bookmark 테이블 (Vibe 보관함)
Column: id / Type: BIGINT / Key: PK / Extra: Auto Increment / 설명: 북마크 식별자
Column: user_id / Type: BIGINT / Key: FK (User.id) / 설명: 참조하는 사용자 식별자
Column: category / Type: VARCHAR(20) / 설명: 콘텐츠 분류 (MOVIE, ANIME, GAME, YOUTUBE)
Column: external_id / Type: VARCHAR(100) / 설명: 외부 API에서 제공하는 해당 콘텐츠의 고유 ID
Column: title / Type: VARCHAR(255) / 설명: 저장 당시의 콘텐츠 제목
Column: thumbnail_url / Type: TEXT / 설명: 저장 당시의 썸네일 이미지 주소
